Progressive Web Apps

Fast forward to 2015 Progressive Web Apps were announced in last year's Chrome Dev summit and everyone went 'ooo'

Progressive Web Apps

  1. discoverable
  2. installable
  3. remain up-to-date
  4. fast
  5. work offline
So what are they? It's an evolution beyond running an app in a browser tab The reason it's called progressive: user experience becomes better as the capabilities of the device improve It's up to you how you implement them but, in general users can get a native app-like experience. even though it's still a web application Visit app first time it downloads all the files it needs to run. Next launch instantaneous. But when you visit a web app for the second time, the browser will prompt you to install it. That adds an icon to your home screen in a single click you don't have to go through the browser menu There's no hunting the AppStore, accepting permissions, installing, waiting then realising an app was rubbish because you've already tried it. When you click this icon, you see a splash screen as your app loads and it launches with whatever browser UI you decide is necessary They're fast. A cold launch little slower than native because it's starting a browser but a warm start can be faster than native which is astounding And finally, they work offline and you have full control over the caching options. OK, so it's not a single technology - nothing is ever that simple let's look at what code you need...

HTTPS

Next, you must deliver your app over HTTPS - not HTTP. The reason for that will become evident soon but it should be a relatively easy step depending on your web host. There is an exception While you're developing, you can access any localhost address and it won't require HTTPS There's also some start-up parameters for Chrome so you can set any URL

Service Worker

And the final piece of the puzzle is a service worker. This is a programmable proxy which sounds horrendously complex it intercepts your application's network calls and can either - return data from the web - or data from the local cache. it's a standard web worker script which means it runs on a separate thread and has no direct access to the page DOM and other objects. It must live in your web root because it can only cache assets in the same folder or lower. And service workers are the reason we need HTTPS. because they decide how network calls is handled Imagine what could happen if a nasty person managed to inject one into your site it would have full control and could serve whatever content they liked.

Service Worker: installation event

self.addEventListener('install', function(e) {

  e.waitUntil(
    caches.open('mycache01').then(function(cache) {
      return cache
        .addAll([
          '/',
          '/index.html',
          '/manifest.json',
          '/css/app.css',
          '/js/app.js',
        ])
        .then(function() { self.skipWaiting(); });
    })
  );

}
The service worker is a single JavaScript file The browser loads it every time you go online or it'll fall back to a cached version. And you should note it's all JavaScript promise based which should make asynchronous calls a little easier (although I'm not always convinced by that) It has three main events you can create handlers for. The first type is an installation event this occurs when the browser loads your app for the first time. Here, we use the CacheStorage API to open a cache named mycache01 and we add the files we need to ensure the app works offline this is known as your application shell and should include the static assets CSS, JavaScript and images. it could also have your basic HTML template or page Note you should include all alternative URLs so we reference a single slash and index.html on lines 7 and 8 even though they're referring to the same file

Service Worker: activation event

self.addEventListener('activate', function(e) {

  e.waitUntil(
    caches.keys().then(function(keyList) {
      return Promise.all(keyList.map(function(key) {
        if (key !== 'mycache01') {
          return caches.delete(key);
        }
      }));
    })
  );

  return self.clients.claim();

});
Next, we have the activation event which runs whenever the application launches. You may not need it. but a typical option is to remove older cached files. This fetches all the caches by name and deletes any which aren't mycache01. So if we ever update the application we could call the cache a different name and the app will re-fetch all static assets.

Service Worker: fetch event

self.addEventListener('fetch', function(e) {

  e.respondWith(
    caches.open('mycache01').then(function(cache) {
      return fetch(e.request).then(function(response) {
        cache.put(e.request.url, response.clone());
        return response;
      });
    })
  );

});
Finally, we have the fetch event which is the trickiest to understand This intercepts file and Ajax requests and responds accordingly. Again, you can do what you like Here we're using the Fetch API which is a simpler promise-based version of XMLHttpRequest and we're just caching Ajax responses when we're online. So our service worker intercepts the request gets the response puts it in the cache then returns it to the main application This code is intentionally simple to show the basics but your application can be different depending on your requirements

app.js: register service worker

// register service worker
if ('serviceWorker' in navigator) {
  navigator.serviceWorker
    .register('/service-worker.js')
    .then(function() { console.log('service worker registered'); });
}
The next step is to load our service worker from our application's main javascript file We can detect the browser supports service workers and register it.

app.js: use cached data

var url = 'https://api.call.io/';

// is data in cache?
if ('caches' in window) {
  caches.match(url).then(function(response) {
    if (response) {
      response.json().then(updateApp);
    }
  });
}

// make new call
var req = new XMLHttpRequest();
req.open('GET', url);
So our service worker just: - cached the static assets and - previous Ajax responses. What we can now do is check whether a response is in the cache before we make an Ajax call we can do that in our app which is not the best way but it keeps things simple lines 5 to 7 pass any cached JSON results to an updateApp function which updates our page. So our app starts instantly when we have cached data But we can then make a real web service call to check if there's any updates and our service worker will cache the result So this caching policy is probably the hardest part of designing your web app deciding what to cache, when to do it and when you use cached data can be tricky. Positive Progressive Points

  • it can enhance your existing web site/app
  • the level of enhancement is up to you
  • Desktop PWA is supported now
  • Use Web app install banner
but there are several positive points - first is that it's an enhancement to your existing system at the very least it'll make your web experience faster and better for users it may even improve your SEO rankings - you don't need full scale redevelopment - you don't need to develop a single page application or anything special - you can just apply it to your existing site and technologies and the level of enhancement is up to you. Google makes numerous recommendations about using material design, fancy animations and making an app look as though it's native but you can ignore all that - the decision is yours you could have an "app" which does nothing more than cache the most popular pages on your web site Google is pushing Progressive Web Apps hard. They announced in August they're dropping Chrome packaged apps in favour of Progressive Web Apps And other browser vendors are behind this too.

…but the same is true for native apps!

Progressive Web Apps finally offer us a reasonable alternative to native apps We'll always need native apps but the distinction between native and the web is increasingly blurred of course, none of this technology makes it easier for us to build apps people actually want to use but that's hard no matter what we do.

Development recommendations

  • try examples
  • put service worker and manifest files in the root
  • test in Chrome 52+
  • use incognito mode
  • test on real devices
so I have a few recommendations which may help service worker code can be confusing Google has a worked sample application which shows how they're developed remember to place your service worker and manifest files in your web root folder use Chrome 52 or above for testing which should be fine because the latest is version 53 the Application tab in developer tools allows you to inspect the manifest and service worker then emulate going offline Incogito mode is really handy for testing your app nothing is cached after you unload the tab. And you can debug on a real device over USB using the "Inspect Devices" menu that's how I managed to install my currency exchange app on a phone There are service worker tools in Firefox They're not part of the standard developer toolset and I struggled with them but better facilities are coming soon.
